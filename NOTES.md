## Розділ 5: Навігація між сторінками

**11. Відображення даних на дашборді**
Отримано дані з бази Postgres за допомогою серверних компонентів (React Server Components) та SQL-запитів. Дані успішно виведені на головній сторінці: статистика в картках, графік доходу та список останніх рахунків.

## Розділ 8: Статичний та динамічний рендеринг

**12. Вікторина: Статичний рендеринг**
Правильна відповідь на запитання про те, чому статичний рендеринг не підходить для дашбордів (бо він не відображатиме останні зміни даних).
![Вікторина](./screenshots/11.png)

**13. Моделювання повільного отримання даних**
Штучно додано 3-секундну затримку у функцію `fetchRevenue()`, щоб продемонструвати проблему: при динамічному рендерингу вся сторінка чекає на завершення найповільнішого запиту.
![Симуляція затримки](./screenshots/12.png)

## Розділ 9: Потокове передавання (Streaming)

**14. Скелет завантаження сторінки**
Створено файл `loading.tsx` у групі маршрутів `(overview)`. Тепер під час завантаження даних користувач бачить пульсуючий скелет `DashboardSkeleton`, що покращує сприйняття швидкості інтерфейсу.
![Dashboard Skeleton](./screenshots/13.png)

**15. Потокове передавання компонентів (Suspense)**
Використано React `Suspense` для незалежного завантаження окремих частин дашборду. Завдяки цьому швидкі компоненти (картки) з'являються одразу, а важкі (графік доходів) — у міру готовності, не блокуючи всю сторінку.
![Suspense Streaming](./screenshots/14.png)

## Розділ 10: Додавання пошуку та пагінації

**15. Реалізація пошуку та пагінації через URL-параметри**
Додано функціонал пошуку рахунків та перемикання сторінок. Замість використання клієнтського стану (React state), стан зберігається в URL-параметрах (`?query=...&page=...`), що дозволяє ділитися посиланнями на результати та виконувати рендеринг на стороні сервера. 
![Search and Pagination](./screenshots/15.png)

## Розділ 11: Мутація даних (Mutating Data)

**16. Форма створення рахунку**
Реалізовано сторінку з формою для створення нового рахунку (`/invoices/create`).
![Create Invoice Form](./screenshots/16.png)

**17. Створення запису через Server Actions**
Написано серверну дію `createInvoice`, яка перевіряє дані за допомогою Zod, зберігає їх у базу даних Postgres та оновлює кеш сторінки за допомогою `revalidatePath`.
![Invoice Created](./screenshots/17.png)

**18. Динамічна маршрутизація та форма редагування**
Створено динамічний маршрут `[id]/edit`. Форма автоматично підтягує існуючі дані рахунку з бази для подальшого редагування.
![Edit Invoice Form](./screenshots/18.png)

**19. Оновлення та видалення записів**
Реалізовано дії `updateInvoice` та `deleteInvoice`. Після збереження змін (або видалення) таблиця миттєво оновлюється без перезавантаження всієї сторінки.
![Invoice Updated](./screenshots/19.png)

## Розділ 12: Обробка помилок (Error Handling)

**20. Обробка непередбачуваних помилок (`error.tsx`)**
Створено файл `error.tsx`, який слугує межею помилок (error boundary) для маршруту. Якщо стається збій (наприклад, помилка бази даних при видаленні), користувач бачить резервний інтерфейс із кнопкою для повторної спроби, а не "зламану" сторінку.
![Generic Error Page](./screenshots/20.png)

**21. Обробка помилок 404 (`not-found.tsx`)**
Реалізовано обробку ситуацій, коли запитуваного ресурсу не існує (наприклад, спроба редагувати рахунок з неіснуючим UUID). Використано функцію `notFound()` та створено кастомний інтерфейс `not-found.tsx` для зручного повернення на попередню сторінку.
![404 Not Found Page](./screenshots/21.png)
